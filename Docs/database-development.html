▶?? MTM Business Logic Rules

✅? IMPLEMENTED CORRECTLY IN NEW PROCEDURES

CRITICAL: TransactionType is determined by USER INTENT, NOT operation numbers.

→IN Transactions

When: User is adding stock to inventory

Procedure: inv_inventory_Add_Item_Enhanced

Example: Receiving new parts, production completion

←OUT Transactions

When: User is removing stock from inventory

Procedure: inv_inventory_Remove_Item_Enhanced

Example: Production consumption, shipment

↔TRANSFER Transactions

When: User is moving stock between locations

Procedure: inv_inventory_Transfer_Item_New

Example: Moving from receiving to production

💡Operation Numbers Are Workflow Steps Only

Operation numbers ("90", "100", "110", etc.) represent manufacturing workflow steps and do NOT determine transaction types:

  • "90" - Raw Material Receipt (workflow step)
  • "100" - Production Processing (workflow step)
  • "110" - Quality Control (workflow step)
  • "120" - Finished Goods (workflow step)

The same operation can be used with any TransactionType depending on what the user is doing.

▶?? Standardized Error Handling

✅All Procedures Include Standard Error Handling

Output Parameters (All Procedures)

  • OUT p_Status INT - Status code
  • OUT p_ErrorMsg VARCHAR(255) - Descriptive message

Status Codes

0 - Success

Operation completed successfully

1 - Warning

Validation failed or business rule violated

-1 - Error

SQL exception or system error occurred

Service Layer Integration Pattern

// Standard error handling for all procedures
public async Task<Result<bool>> AddInventoryItemAsync(InventoryItem item)
{
    try
    {
        var result = await Helper_Database_StoredProcedure.ExecuteDataTableWithStatus(
            Model_AppVariables.ConnectionString,
            "inv_inventory_Add_Item_Enhanced",
            parameters
        );

        if (result.Status == 0)
        {
            return Result<bool>.Success(true, result.Message);
        }
        else if (result.Status == 1)
        {
            return Result<bool>.Warning(false, result.Message);
        }
        else
        {
            return Result<bool>.Error(result.Message);
        }
    }
    catch (Exception ex)
    {
        return Result<bool>.Error($"Service layer error: {ex.Message}");
    }
}

                ["p_ReferenceNumber"] = referenceNumber,
                ["p_Notes"] = notes,
                ["p_UserID"] = userId ?? "system"

            var result = await Helper_Database_StoredProcedure.ExecuteDataTableWithStatus(
                _connectionString,
                "inv_inventory_Add_Item_Enhanced",
                parameters

                ? Result<bool>.Success(true, result.Message)
                : Result<bool>.Warning(false, result.Message);

[Test]
public async Task AddInventory_ShouldCreateINTransaction_WhenUserAddsStock()
{
    // Arrange
    var partId = "TEST001";
    var operationId = "90"; // Workflow step number
    var locationId = "RECEIVING";
    var quantity = 100;
    var userId = "testuser";
    
    // Act - Adding stock should create IN transaction (user intent)
    var result = await _inventoryService.AddInventoryAsync(
        partId, operationId, locationId, quantity, userId: userId);
    
    // Assert
    Assert.True(result.IsSuccess);
    Assert.Contains("Successfully added", result.Message);
    
    // Verify transaction type is IN regardless of operation number
    var transactions = await _transactionService.GetTransactionHistoryAsync(partId, userId);
    Assert.AreEqual("IN", transactions.Value.First().TransactionType);
    Assert.AreEqual(operationId, transactions.Value.First().OperationId);
}
