CoreModels

The MTM Data Foundation

🎯 What This Is

The Digital Blueprint Library

Think of CoreModels as your digital blueprint library - like having detailed engineering drawings and specifications for every piece of equipment, process, and record in your manufacturing facility. These models define exactly how data is structured, stored, and used throughout the entire MTM WIP system.

🏭 Manufacturing Floor Analogy

Imagine you have a comprehensive documentation center that contains:

  • 👥 Employee records and roles - Who can do what in each work area
  • 📦 Part specifications - Every component, where it goes, and how it's tracked
  • 📊 Transaction logs - Complete history of every item movement
  • 🏭 Work center definitions - All locations and operation numbers
  • ⚡ Quick action templates - Preset configurations for common operations
  • 📝 Error and audit logs - Complete tracking of all system events

🔑 Key Business Models

📦 InventoryItem

What's currently in your warehouse

  • PartID - Which part this is (string format)
  • Location - Where it's stored (work center code)
  • Operation - Which workflow step (string number like "90", "100")
  • Quantity - How many pieces
  • User - Who last handled it
  • BatchNumber - Group tracking identifier

📊 InventoryTransaction

Record of every inventory movement

  • TransactionType - IN, OUT, or TRANSFER
  • PartID - Which part was moved
  • FromLocation/ToLocation - Movement path
  • Operation - Workflow step number
  • Quantity - How many were moved
  • User - Who performed the action

👤 User

Employee information and settings

  • User_Name - Login identifier
  • Shift - Which work shift (1, 2, 3)
  • Theme preferences - Visual appearance settings
  • Database connection - Server and port configuration
  • VitsUser - Special permissions flag

⚡ QuickTransactionButton

User's top 10 frequent operations

  • User - Which employee this belongs to
  • PartID - Most used part
  • Operation - Most used workflow step
  • Quantity - Typical quantity for this operation
  • Position - Button position (1-10)

📋 MTM Business Rules

🚨 Critical Data Rules

  • Part IDs are strings - Always text format (e.g., "PART001")
  • Operations are string numbers - Workflow steps like "90", "100", "110"
  • Quantities are integers - Whole numbers only, no decimals
  • TransactionType - Only IN, OUT, or TRANSFER allowed
  • UI positions use 1-based indexing - Positions start at 1, not 0

⚠️ Data Consistency Rules

  • User tracking - Every transaction must record which user performed it
  • Timestamp recording - All operations must have date/time stamps
  • Location validation - Locations must exist in master data
  • Operation validation - Operations must be valid workflow steps
  • Batch tracking - Related items grouped by batch numbers

ℹ️ Compatibility Guidelines

  • Multiple property names - Models support different naming conventions
  • Database mapping - Properties map to specific database table columns
  • JSON serialization - Models can be converted to/from JSON format
  • Navigation properties - Related data can be loaded together

🔄 Transaction Types

📥 IN Transactions

Adding items to inventory

  • Receiving new parts from suppliers
  • Completing manufacturing operations
  • Returning items to stock
  • Adjusting quantities upward
Example: Received 50 units of PART001 at operation 90 in work center WC01

📤 OUT Transactions

Removing items from inventory

  • Shipping completed parts to customers
  • Consuming raw materials in production
  • Scrapping defective items
  • Adjusting quantities downward
Example: Removed 25 units of PART002 from operation 100 in work center WC02

🔄 TRANSFER Transactions

Moving items between locations

  • Moving parts between work centers
  • Advancing through workflow operations
  • Relocating for storage optimization
  • Balancing inventory across locations
Example: Transferred 15 units of PART003 from WC01 to WC03
