🛠GitHub Workflow

🚧 Important: All commits should include compliance verification for changed files. No merges until core systems are implemented.
▶Development Workflow
  1. Pre-Development Check
    • Verify missing systems implementation status
    • Check current compliance metrics (must be >80% for new development)
    • Review implementation roadmap and priority order
  2. Branch Creation
    • Create feature branch: feature/implement-{system-name}
    • For missing systems: feature/phase-{n}-{system-name}
    • For compliance fixes: fix/compliance-{area}
  3. Development
    • Follow phase-based implementation order
    • Use custom prompts for consistent implementation
    • Generate compliance reports for new code
  4. Pre-Commit Verification
    • Generate compliance report for changed files
    • Address critical and high-priority violations
    • Verify no new missing systems introduced
    • Run build verification
  5. Pull Request
    • Include compliance status in PR description
    • Reference implementation phase and custom prompts used
    • Attach compliance reports for reviewer assessment
▶Commit Message Standards
  • Missing Systems Implementation:
    • feat: implement Result<T> pattern system (Phase 1)
    • feat: add Models namespace foundation (Phase 1)
    • feat: setup dependency injection container (Phase 1)
  • Compliance Fixes:
    • fix: replace hard-coded colors with DynamicResource
    • fix: implement ReactiveUI patterns in InventoryViewModel
    • fix: add Service_ErrorHandler integration
  • Quality Assurance:
    • docs: add compliance report for InventoryService
    • test: add unit tests for service layer
    • chore: update compliance metrics dashboard
▶Quality Gates

Development Blocking Issues

The following violations should BLOCK merges:

  • Missing service layer implementation
  • Missing data models causing type safety issues
  • Missing DI container preventing service injection
  • Direct view instantiation violating MVVM
  • Hard-coded colors preventing theming
  • Compliance score below 80% for foundational systems

Pre-Merge Checklist

  • Compliance report generated and reviewed
  • Critical violations addressed
  • Phase-based implementation order followed
  • Custom prompts used for consistency
  • Build verification successful
  • No new missing systems introduced
  • Documentation updated if needed
